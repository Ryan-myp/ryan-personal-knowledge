## 第一部分：分布式系统核心概念

### CAP 定理

```
CAP 三选一：
┌─────────────────────────────────────────────────────────────────────┐
│ Consistency（一致性）：所有节点数据同步                               │
│ Availability（可用性）：每次请求都能得到响应                          │
│ Partition Tolerance（分区容错）：网络分区时系统仍能工作               │
│                                                                     │
│ 实际选择：                                                          │
│ • CP：数据库（MySQL、ZooKeeper）— 一致性优先                          │
│ • AP：缓存（Redis、Cassandra）— 可用性优先                           │
│ • CA：单机系统（理论上不存在）                                        │
│                                                                     │
│ 分布式系统 = 必须容忍分区（P），所以只能在 C 和 A 之间权衡            │
└─────────────────────────────────────────────────────────────────────┘
```

### 一致性模型

```
一致性模型对比：
┌────────────────┬────────────┬────────────┬────────────┐
│     模型       │  延迟      │  可用性    │  适用场景  │
├────────────────┼────────────┼────────────┼────────────┤
│ 强一致性       │ 高        │ 低        │ 金融交易   │
│ 最终一致性     │ 中        │ 高        │ 大多数场景 │
│ 因果一致性     │ 低        │ 高        │ 社交网络   │
│ 会话一致性     │ 低        │ 高        │ Web 应用   │
└────────────────┴────────────┴────────────┴────────────┘
```

## 第二部分：分布式一致性协议

### Paxos 算法

```
Paxos 两阶段提交：
1. Prepare 阶段：Proposer 提议序号，询问 Acceptor
2. Accept 阶段：Acceptor 承诺，Proposer 发送值

关键概念：
• Proposer：提议者
• Acceptor：接受者
• Learner：学习者

容错能力：容忍 N/2 个节点故障
```

### Raft 算法

```
Raft 状态机：
┌────────────┬────────────┬────────────┐
│   Follower │   Candidate│   Leader   │
├────────────┼────────────┼────────────┤
│ 被动接收   │ 竞选投票   │ 管理复制   │
│ 超时→Candidate │        │            │
│            │ 赢得选举→Leader │            │
└────────────┴────────────┴────────────┘

Raft 日志复制：
1. Leader 接收客户端请求
2. Leader 追加到日志
3. Leader 复制日志到 Follower
4. 多数派确认后提交
5. Leader 通知客户端

优势：比 Paxos 更容易理解和实现
```

## 第三部分：分布式系统设计模式

### 1. 分片（Sharding）

```
分片策略：
1. 范围分片：按 key 范围划分
2. 哈希分片：hash(key) % N
3. 一致性哈希：减少迁移

广告场景：
• 用户分片：user_id % 100
• 广告分片：ad_id % 50
• 按地域分片：PH/ID/MY/TH
```

### 2. 复制（Replication）

```
复制模式：
1. 主从复制：一主多从，写主读从
2. 多主复制：多主写入，冲突解决
3. 对等复制：所有节点平等

广告场景：
• 主从：广告数据写主，查询读从
• 对等：用户画像多副本
```

### 3. 缓存（Caching）

```
缓存策略：
1. Cache-Aside：应用管理缓存
2. Read-Through：缓存代理读取
3. Write-Through：缓存代理写入
4. Write-Behind：异步写入

广告场景：
• Redis：用户画像、实时特征
• 本地缓存：配置、字典
• CDN：创意素材
```

## 第四部分：微服务架构

### 服务通信

```
通信方式：
1. REST API：HTTP + JSON，简单通用
2. gRPC：HTTP/2 + Protobuf，高性能
3. 消息队列：异步解耦

广告场景：
• 同步：竞价请求、用户查询
• 异步：事件通知、日志收集
```

### 服务发现

```
服务发现模式：
1. 客户端发现：Consul、Eureka
2. 服务器发现：Nginx、HAProxy
3. DNS 发现：Kubernetes Service

广告场景：
• Kubernetes Service：内部服务发现
• Consul：跨集群服务发现
```

### 熔断与降级

```
熔断模式：
1. 熔断器：Open/Closed/HalfOpen
2. 超时控制：请求超时 + 连接超时
3. 降级策略：缓存、默认值、空结果

广告场景：
• 竞价服务：超时 50ms，失败降级
• 排序服务：模型不可用，用规则
```

## 第五部分：分布式事务

### 事务模式

```
分布式事务方案：
1. 2PC（两阶段提交）：强一致性，性能差
2. TCC（Try-Confirm-Cancel）：应用层控制
3. Saga：长事务，逐步补偿
4. 本地消息表：最终一致性

广告场景：
• 预算扣减：TCC（强一致性）
• 广告发布：Saga（最终一致性）
• 数据同步：本地消息表（异步）
```

### Saga 实现

```
Saga 模式：
┌─────────────────────────────────────────────────────────────────────┐
│ Step 1: 创建广告 → 补偿：删除广告                                    │
│ Step 2: 扣减预算 → 补偿：恢复预算                                    │
│ Step 3: 上传素材 → 补偿：删除素材                                    │
│ Step 4: 发布广告 → 补偿：下架广告                                    │
│                                                                     │
│ 协调器：                                                              │
│ 1. 顺序执行正向操作                                                  │
│ 2. 任何一步失败，执行已执行步骤的补偿                                │
│ 3. 全部成功后提交事务                                                │
└─────────────────────────────────────────────────────────────────────┘
```

## 第六部分：自测题

### Q1: CAP 为什么不能全选？

**A**: 网络分区不可避免，必须选 P，所以只能在 C 和 A 之间权衡。

### Q2: Raft 和 Paxos 的区别？

**A**: Raft 更易理解实现，Paxos 更通用。Raft 有明确的 Leader。

### Q3: 广告系统用什么一致性模型？

**A**: 预算扣减用强一致性（TCC），查询用最终一致性（缓存）。
