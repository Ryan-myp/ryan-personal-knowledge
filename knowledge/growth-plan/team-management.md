### 2.2 面试流程设计

```
面试流程 (4 轮):

┌──────────────────────────────────────────────────────────────┐
│                                                              │
│  Round 1: 技术面 (45min)                                     │

│    - 性能调优经验                                           │
│  • 编码题 (15min)                                            │
│    - 一道中等难度的算法题                                    │
│    - 重点: 思路 + 代码质量, 不是速度                         │
│                                                              │
│  Round 2: 系统设计 (60min)                                   │
│  ────────────────────                                        │
│  • 开放系统设计 (40min)                                      │
│    - 设计一个高并发的广告系统                                │
│    - 评估: 架构合理性/扩展性/容错/性能                      │
│  • 深入追问 (20min)                                          │
│    - 针对候选人的方案深入挖掘                               │
│    - 评估: 思考深度/技术视野                                 │
│                                                              │
│  Round 3: 工程实战 (45min)                                   │
│  ────────────────────                                        │
│  • Code Review (20min)                                       │
│    - 给一段有问题的代码, 让候选人做 CR                        │
│    - 评估: 问题发现能力/代码标准                             │
│  • 排障 (25min)                                              │
│    - 给一个线上故障场景, 让候选人排查                        │
│    - 评估: 排障思路/优先级判断                               │
│                                                              │
│  Round 4: 综合面 (30min)                                     │
│  ────────────────────                                        │
│  • 团队协作与文化匹配                                        │
│  • 职业发展期望                                              │
│  • TL 1:1 面谈                                               │
│                                                              │
│  评估标准 (Score Card):                                      │
│  ┌──────────────────────────────────────────────────────────┐
│  │ 维度            │ 1 (弱)   │ 2 (一般) │ 3 (合格) │ 4 (强) │
│  ├──────────────────────────────────────────────────────────┤
│  │ Go 语言         │ 基础     │ 熟练     │ 深入     │ 专家   │
│  │ 系统设计        │ 简单     │ 常规     │ 复杂     │ 大规模 │
│  │ 编码质量        │ 功能     │ 可读     │ 优雅     │ 卓越   │
│  │ 工程素养        │ 缺少     │ 有         │ 熟练     │ 精通   │
│  │ 协作沟通        │ 被动     │ 正常     │ 积极     │ 引领   │
│  └──────────────────────────────────────────────────────────┘
│  录用标准: 总分 ≥ 3.0 且无维度 < 2                             │
└──────────────────────────────────────────────────────────────┘
```

### 3.1 个人发展计划 (PDP)

```go
// pdp.go — 个人发展计划
package management

import (
	"time"
)

// PDP 个人发展计划
type PDP struct {
	EmployeeID   string    `json:"employee_id"`
	Name         string    `json:"name"`
	CurrentLevel string    `json:"current_level"`   // Junior/Mid/Senior
	TargetLevel  string    `json:"target_level"`    // 目标级别
	Quarter      string    `json:"quarter"`         // Q3 2025
	StartDate    time.Time `json:"start_date"`
	EndDate      time.Time `json:"end_date"`
	
	// 成长目标
	TechnicalGoals []GrowthGoal `json:"technical_goals"`      // 技术目标
	LeadershipGoals []GrowthGoal `json:"leadership_goals"`     // 领导力目标
	BusinessGoals  []GrowthGoal `json:"business_goals"`       // 业务目标
	
	// 行动计划
	Actions []GrowthAction `json:"actions"` // 具体行动
	
	// 里程碑
	Milestones []Milestone `json:"milestones"` // 阶段里程碑
	
	// 跟踪
	ProgressReports []ProgressReport `json:"progress_reports"`
	
	// 评估
	FinalAssessment string `json:"final_assessment"`
}

// GrowthGoal 成长目标
type GrowthGoal struct {
	Title       string   `json:"title"`       // 目标标题
	Description string   `json:"description"` // 详细描述
	Measure     string   `json:"measure"`     // 衡量标准
	DueDate     string   `json:"due_date"`    // 截止日期
	Priority    string   `json:"priority"`    // P0/P1/P2
	Status      string   `json:"status"`      // NotStarted/InProgress/Completed
}

// GrowthAction 成长行动
type GrowthAction struct {
	Type       string `json:"type"`       // 学习/实践/分享/ mentoring
	Content    string `json:"content"`    // 具体内容
	Owner      string `json:"owner"`      // 执行者 (自己/mentor/TL)
	DueDate    string `json:"due_date"`
	Milestone  string `json:"milestone"`  // 对应的里程碑
}

// 成长路径设计 (6-12 个月):
//
// Junior → Mid (3-6个月):
// ──────────────────────
// 技术: Go 语言中级 → 独立负责模块
// 工程: 单元测试, Code Review, 基础排障
// 协作: 参与团队协作, 能清晰表达技术观点
// 业务: 理解广告系统基本流程
//
// Mid → Senior (6-12个月):
// ──────────────────────
// 技术: 深入源码 (Redis/MySQL/GMP), 架构设计能力
// 工程: 性能调优, 系统设计, 带新人
// 协作: 主导技术项目, 跨团队协作
// 业务: 深入广告业务, 能独立设计方案
//
// Senior → 高级专家 (12-18个月):
// ────────────────────────────
// 技术: 技术深度 + 广度, 行业前沿
// 工程: 架构演进, 技术决策
// 协作: 技术领导力, 团队建设
// 业务: 技术驱动业务, ROI 意识

// PDP 模板 (每个季度):
// 员工: [姓名]
// 当前级别: [Junior/Mid/Senior]
// 目标级别: [Mid/Senior]
// 季度: Q3 2025
//

// ├── [P0] 掌握 Go 源码级 GMP 调度原理
```

### 3.2 1:1 面谈指南

```
1:1 面谈 (每周 30min):

┌──────────────────────────────────────────────────────────────┐
│                                                              │
│  1:1 面谈三大主题:                                           │
│  ──────────────────                                           │
│                                                              │
│  1. 工作进展 (Work Progress) — 10min                         │
│  ──────────────────────────                                   │
│  • 当前任务进展                                              │
│  • 遇到的阻塞/困难                                           │
│  • 需要资源/支持                                             │
│  • 下周计划                                                  │
│                                                              │
│  2. 成长与发展 (Growth & Development) — 15min                │
│  ─────────────────────────────────────────                    │
│  • PDP 进展回顾                                              │
│  • 技术成长方向                                              │
│  • 导师反馈                                                 │
│  • 职业发展规划                                              │
│                                                              │
│  3. 反馈与认可 (Feedback & Recognition) — 5min               │
│  ─────────────────────────────────────────                    │
│  • 本周做得好的 (具体事例)                                   │
│  • 改进建议 (建设性反馈)                                     │
│  • 团队文化建设                                              │
│                                                              │
│  关键技巧:                                                   │
│  ──────────                                                   │
│  • 70% 倾听, 30% 说                                       │
│  • 关注 "人" 而不仅是 "事"                                  │
│  • 用 SBI 模型给反馈 (Situation-Behavior-Impact)             │
│  • 记录行动项, 下次 1:1 跟进                                │
│  • 保密原则: 1:1 内容是保密的                                │
│                                                              │
│  SBI 反馈模型:                                               │
│  ───────────────────                                           │
│  S (Situation) 情境: "在昨天 code review 中..."             │
│  B (Behavior) 行为: "...你使用了大量 Java 风格的泛型写法"   │
│  I (Impact) 影响: "...导致 Go 同事理解困难, 延长 review 时间"│
│  → 而不是: "你代码写得太差了!"                              │
└──────────────────────────────────────────────────────────────┘
```

```
冲突解决五步法:

1. 识别冲突 (Identify)
   ├── 技术分歧 (方案 A vs 方案 B)
   ├── 资源冲突 (争夺服务器/人力)
   ├── 进度冲突 (排期不一致)
   └── 人际冲突 (沟通风格/期望不一致)

2. 理解双方 (Understand)
   ├── 各自立场背后的利益
   ├── 数据和事实依据
   └── 情绪因素

3. 寻找共同点 (Common Ground)
   ├── 共同目标 (用户价值/业务目标)
   ├── 共同约束 (预算/时间/技术)
   └── 共同价值观 (代码质量/用户体验)

4. 制定方案 (Generate Solutions)
   ├── 头脑风暴 (多个选项)
   ├── 评估方案 (Pros/Cons)
   └── 选择最优 (或折中方案)

5. 执行与跟进 (Execute & Follow-up)
   ├── 明确行动项 (谁/何时/做什么)
   ├── 定期跟进 (检查进展)
   └── 持续改进 (总结经验)

冲突解决原则:
- 对事不对人
- 数据驱动决策
- TL 做最终决策者 (但尊重专业意见)
- 决策后全力执行 (Discipline and Execute)
```

### Q1：如何平衡技术深度和管理时间？

<details>
<summary>点击查看参考答案</summary>

**核心原则**: "技术深度是你的根基, 但管理是你的杠杆。"

**时间分配建议**:
- 编码/技术: 20-30% (保持技术手感)
- 代码审查: 15-20% (质量把关)
- 团队管理: 30-40% (1:1/绩效/招聘/发展)
- 跨团队协作: 15-20%
- 个人学习: 10-15%

**保持技术深度的方法**:
1. 深度代码审查 (不只是看, 是思考)
2. 参与关键架构决策
3. 定期技术分享/写作
4. 每周至少 2 小时深入学习
5. 参与开源/技术社区

**避免的陷阱**:
- ❌ 完全脱离编码 → 技术判断力下降
- ❌ 只写代码不管理 → 团队效能低
- ❌ 过度管理不学习 → 技术脱节
</details>

### Q2：如何评估一个中级工程师是否准备好晋升 Senior？

<details>
<summary>点击查看参考答案</summary>

**四个维度的 Senior 标准**:

1. **技术深度**: 能独立设计复杂系统, 源码级理解关键组件
2. **交付质量**: 持续交付高质量代码, 低 Bug 率
3. **领导力**: 能带新人, 做技术分享, 主导小型项目
4. **业务意识**: 理解业务目标, 能用技术驱动业务

**评估方法**:
- 过去一年主导过至少 2 个复杂项目
- 代码审查能发现深层次问题
- 技术分享获得团队认可
- 能独立排障线上复杂问题
- PDP 目标全部达成

**面试评估**:
- 系统设计: 能设计可扩展/容错的系统
- 编码: 代码质量/测试意识/性能考虑
- 行为面: 领导力和协作表现
</details>

### Q3：PDP 制定中常见的错误是什么？

<details>
<summary>点击查看参考答案</summary>

**常见错误**:

1. 目标太模糊 → "提高技术能力" → 改为 "掌握 Go 源码级 GMP 原理"
2. 目标太多 → 每个季度最多 3-4 个核心目标
3. 缺乏衡量标准 → "学习新技术" → 改为 "完成 X 项目, 产出 Y 文档"
4. 不考虑个人兴趣 → 结合个人兴趣和业务需求
5. 没有行动项 → 每个目标有具体的行动和里程碑
6. 没有跟进 → 每两周检查 PDP 进展
7. TL 单方面制定 → 和员工共同讨论制定
</details>

### 2.1 人才画像 (Person Profile)

```go
// talent_profile.go — 人才画像模型
package management

import (
	"fmt"
	"time"
)

// SkillLevel 技能等级
type SkillLevel int

const (
	SkillJunior  SkillLevel = 1 // 初级
	SkillMid     SkillLevel = 2 // 中级
	SkillSenior  SkillLevel = 3 // 高级
	SkillStaff   SkillLevel = 4 // 专家
	SkillPrincipal SkillLevel = 5 // 资深专家
)

// SkillCategory 技能类别
type SkillCategory string

const (
	SkillTechnical   SkillCategory = "technical"    // 技术能力
	SkillArchitecture SkillCategory = "architecture" // 架构能力
	SkillLeadership  SkillCategory = "leadership"   // 领导力
	SkillBusiness    SkillCategory = "business"     // 业务理解
	SkillCommunication SkillCategory = "communication" // 沟通能力
)

// Skill 技能项
type Skill struct {
	Name    string        `json:"name"`
	Category SkillCategory `json:"category"`
	Level   SkillLevel    `json:"level"`
	Evidence string       `json:"evidence"` // 技能证据
}

// PersonProfile 人才画像
type PersonProfile struct {
	Name        string      `json:"name"`
	Email       string      `json:"email"`
	Role        string      `json:"role"` // 工程师/TL/Lead
	Level       SkillLevel  `json:"level"`
	Skills      []Skill     `json:"skills"`
	Achievements []string   `json:"achievements"`
	Strengths   []string    `json:"strengths"`    // 优势
	Weaknesses  []string    `json:"weaknesses"`   // 待改进
	Goals       []string    `json:"goals"`        // 个人目标
	Availability int        `json:"availability"` // 可投入新项目时间 (小时/周)
	LastReview  time.Time   `json:"last_review"`  // 最近一次绩效评估
	Notes       string      `json:"notes"`        // 备注
}

// TeamMatrix 团队技能矩阵
type TeamMatrix struct {
	Members  []*PersonProfile
	Skills   []string   // 所有技能名称
	Levels   []SkillLevel // 所有等级
}

// BuildTeamMatrix 构建团队技能矩阵
func (m *TeamMatrix) BuildTeamMatrix(team []*PersonProfile) {
	m.Members = team
	
	// 收集所有技能
	allSkills := make(map[string]bool)
	for _, member := range team {
		for _, skill := range member.Skills {
			allSkills[skill.Name] = true
		}
	}
	
	for skill := range allSkills {
		m.Skills = append(m.Skills, skill)
	}
}

// GetSkillDistribution 获取技能分布
func (m *TeamMatrix) GetSkillDistribution() map[string]map[int]int {
	dist := make(map[string]map[int]int)
	
	for _, skillName := range m.Skills {
		dist[skillName] = make(map[int]int)
		for _, member := range m.Members {
			for _, skill := range member.Skills {
				if skill.Name == skillName {
					dist[skillName][int(skill.Level)]++
				}
			}
		}
	}
	
	return dist
}
```

## 第六部分：自测

### Q1：1:1 沟通中，TL 应该如何处理团队成员的压力问题？

<details>
<summary>点击查看参考答案</summary>

**处理步骤**:

1. **倾听为主**: 先让成员表达, 不打断, 不评判
2. **共情**: 表示理解, "我能理解你的感受"
3. **分析根源**: 是工作量大? 技术问题? 人际冲突? 个人问题?
4. **制定行动计划**:
   - 工作量大 → 重新分配任务/增加资源
   - 技术问题 → 提供指导/Mentoring
   - 人际冲突 → 调解/调整协作方式
   - 个人问题 → 提供弹性/资源支持
5. **跟进**: 下周 1:1 检查改善情况

**禁忌**:
- ❌ 忽视或淡化压力 ("别想太多")
- ❌ 立即给解决方案 (先倾听)
- ❌ 泄露隐私 (与其他同事讨论)
- ❌ 承诺做不到的事情

**最佳实践**:
- 建立信任关系 (定期 1:1)
- 关注心理健康 (EAP 资源)
- 营造安全的工作环境
</details>

<details>
<summary>点击查看参考答案</summary>

**判断标准 (四维度)**:

1. **业绩**: 过去 6 个月持续达到或超过当前 level 的要求
2. **能力**: 已具备下一 level 的核心能力 (通过评估)
3. **影响**: 已经在下一 level 的角色中发挥作用 (Mentoring/系统设计)
4. **文化**: 符合团队价值观, 团队协作良好

**晋升流程**:
1. TL 提名 (基于评估)
2. 技能矩阵对比 (与目标 level 对比)
3. 绩效评估 (过去 2-3 次 1:1 + 季度评估)
4. 跨级评审 (其他 TL/Manager 评审)
5. HR 审批
6. 晋升通知

**关键**: 晋升不是奖励, 是角色转变。要评估候选人是否已经在新角色中表现。
</details>

### Q3：OKR 和 KPI 有什么区别？什么时候用哪个？

<details>
<summary>点击查看参考答案</summary>

**OKR (Objectives & Key Results)**:
- 方向性的, 鼓舞人心的目标
- Key Results 是衡量目标是否达成的关键结果
- 适合创新/探索性工作
- 通常不设奖惩 (避免保守)
- 季度制定, 月度 Review

**KPI (Key Performance Indicators)**:
- 量化的, 可衡量的指标
- 适合成熟/可预测的工作
- 通常与绩效/奖惩挂钩
- 年度或半年度制定

**你的场景建议**:
- TL 角色 → OKR 为主 (技术路线图/团队建设)
- 工程师个体 → OKR 为主 (个人成长/项目交付)
- 团队指标 → OKR + KPI 混合 (团队交付 + 性能指标)

**最佳实践**: 用 OKR 设定方向, 用 KPI 跟踪日常运营。
</details>
