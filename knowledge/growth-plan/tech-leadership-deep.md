### 3.2 技术债量化 (技术债度量模型)

```go
// tech_debt.go — 技术债量化模型 (Go 实现)
package leadership

import (
	"fmt"
	"strings"
	"time"
)

// DebtType 技术债类型
type DebtType string

const (
	DebtCodeQuality   DebtType = "code_quality"    // 代码质量
	DebtDocumentation  DebtType = "documentation"   // 文档缺失
	DebtTesting        DebtType = "testing"         // 测试不足
	DebtArchitecture   DebtType = "architecture"    // 架构问题
	DebtSecurity       DebtType = "security"        // 安全风险
	DebtPerformance    DebtType = "performance"     // 性能问题
	DebtScalability    DebtType = "scalability"     // 扩展性问题
	DebtCompliance     DebtType = "compliance"      // 合规问题
)

// TechnicalDebt 技术债项
type TechnicalDebt struct {
	ID          string     `json:"id"`
	Title       string     `json:"title"`
	Type        DebtType   `json:"type"`
	Description string     `json:"description"`
	Priority    DebtPriority `json:"priority"` // critical/high/medium/low
	RiskScore   float64    `json:"risk_score"` // 0-10
	ImpactScore float64    `json:"impact_score"` // 0-10
	CostScore   float64    `json:"cost_score"`   // 0-10: 修复成本 (越大越贵)
	AgeMonths   int        `json:"age_months"`   // 存在时间 (月)
	Owner       string     `json:"owner"`
	Status      string     `json:"status"` // open/in_progress/resolved
	CreatedAt   time.Time  `json:"created_at"`
	UpdatedAt   time.Time  `json:"updated_at"`
}

// DebtPriority 技术债优先级
type DebtPriority string

const (
	PriorityCritical DebtPriority = "critical" // 立即修复
	PriorityHigh     DebtPriority = "high"     // 下一个迭代
	PriorityMedium   DebtPriority = "medium"   // 本季度
	PriorityLow      DebtPriority = "low"      // 有空再做
)

// DebtRiskScore 技术债风险评分 (0-100)
// 公式: Risk = RiskScore × (1 + AgeMultiplier) × ImpactMultiplier
// RiskScore = 0.3 × Risk + 0.4 × Impact + 0.3 × Cost
func (d *TechnicalDebt) CalculateRiskScore() float64 {
	baseRisk := 0.3*d.RiskScore + 0.4*d.ImpactScore + 0.3*d.CostScore
	ageMultiplier := 1.0 + float64(d.AgeMonths)/12.0 // 每过一年增加 100%
	return baseRisk * ageMultiplier
}

// DebtManager 技术债管理器
type DebtManager struct {
	debtList []*TechnicalDebt
}

// AddDebt 添加技术债项
func (m *DebtManager) AddDebt(debt *TechnicalDebt) {
	m.debtList = append(m.debtList, debt)
}

// GetTopDebt 获取风险最高的 N 个技术债
func (m *DebtManager) GetTopDebt(n int) []*TechnicalDebt {
	// 计算所有技术债的风险评分
	type debtWithRisk struct {
		debt    *TechnicalDebt
		risk    float64
	}
	
	type debtWithRisk {
		debt    *TechnicalDebt
		risk    float64
	}
	
	var scored []*debtWithRisk
	for _, debt := range m.debtList {
		risk := debt.CalculateRiskScore()
		scored = append(scored, &debtWithRisk{debt, risk})
	}
	
	// 按风险评分降序排序
	sort.Slice(scored, func(i, j int) bool {
		return scored[i].risk > scored[j].risk
	})
	
	// 返回前 N 个
	if n > len(scored) {
		n = len(scored)
	}
	
	var result []*TechnicalDebt
	for i := 0; i < n; i++ {
		result = append(result, scored[i].debt)
	}
	
	return result
}

// GetDebtByType 按类型分组统计
func (m *DebtManager) GetDebtByType() map[DebtType]int {
	counts := make(map[DebtType]int)
	for _, debt := range m.debtList {
		counts[debt.Type]++
	}
	return counts
}
```

## 第五部分：自测

### Q1：技术债的 "Reckless" 和 "Pragmatic" 有什么区别？

<details>
<summary>点击查看参考答案</summary>

**Reckless ( recklessness ) 债**:
- 明知有问题但不处理
- 没有计划, 没有文档, 没有优先级
- 团队默认 "以后再说"
- 风险: 高 (随时可能崩溃)
- 处理: 立即列入最高优先级

**Pragmatic (pragmatic) 债**:
- 权衡后的选择, 有意为之
- 有明确的偿还计划
- 用于快速获取 MVP
- 风险: 低 (明确计划偿还)
- 处理: 按计划偿还

**关键区别**: Pragmatic 是 "有计划的债", Reckless 是 "无计划的债"。
</details>

### Q2：ADR 的核心价值是什么？为什么不能只在口头或 Slack 中讨论架构决策？

<details>
<summary>点击查看参考答案</summary>

**ADR 的核心价值**:

1. **可追溯性**: 为什么做了这个决策? 谁做的? 什么时候?
2. **知识沉淀**: 新成员能快速了解历史决策
3. **责任明确**: 决策者明确, 避免事后推诿
4. **上下文完整**: 包含替代方案, 未来回顾时有完整视角
5. **决策质量**: 书面化倒逼思考更充分

**口头/Slack 的局限**:
- 信息散乱, 难以搜索
- 无结构化, 缺少上下文
- 无法追踪决策状态
- 团队成员理解不一致

**最佳实践**:
- 重要决策必须写 ADR
- ADR 存入版本控制系统 (Git)
- Review 流程中检查是否有 ADR
- 定期 Review 已接受的 ADR 是否仍然适用
</details>

### Q3：技术债管理的 "20% 法则" 是什么意思？

<details>
<summary>点击查看参考答案</summary>

**20% 法则**: 每个 Sprint 至少预留 20% 的容量用于技术债偿还。

**为什么是 20%**:
- 低于 20%: 技术债会加速积累
- 高于 20%: 可能影响业务交付
- 20% 是一个经验值, 可根据团队情况调整

**实施方法**:
1. 在每个 Sprint Planning 中预留 20% 容量
2. 技术债项按优先级排序
3. 优先处理 Critical 和 High 优先级
4. 每月 Review 技术债偿还进度
5. 调整 20% 比例 (根据业务压力和技术债积累速度)

**注意**: 20% 是底线, 如果技术债严重, 可能需要更高比例。
</details>
