# 架构设计方法论：从需求到架构的完整流程

> 架构设计原则 + 需求分析 + 技术选型 + 架构演进 + 生产排障

---

## 第一部分：架构设计方法论（5 分钟速览）

### 类比理解架构设计

```
盖房子 vs 搭积木：

传统开发 = 搭积木
  → 一块一块拼起来
  → 结构简单，但扩展性差

架构设计 = 盖房子
  → 先打地基（基础设施）
  → 再建框架（核心架构）
  → 最后装修（功能实现）
  → 要考虑抗震（高可用）、扩建（可扩展）
```

### 架构设计的核心问题

```
1. 需求是什么？（功能需求 + 非功能需求）
2. 规模有多大？（QPS/数据量/用户数）
3. 约束条件？（时间/预算/技术栈/团队）
4. 权衡取舍？（一致性 vs 可用性 vs 性能）
5. 如何演进？（从单体到微服务）
```

## 第二部分：需求分析

### 2.1 功能需求 vs 非功能需求

```
功能需求（Functional Requirements）：
- 用户能做什么？
- 系统能提供什么功能？

非功能需求（Non-Functional Requirements）：
- 系统有多快？（性能）
- 系统有多可靠？（可用性）
- 系统能支撑多少用户？（扩展性）
- 系统有多安全？（安全性）
- 系统有多好维护？（可维护性）
```

### 2.2 广告平台需求分析

```
功能需求：
1. 广告主创建广告活动
2. 用户浏览广告
3. 实时竞价
4. 广告展示
5. 点击/转化追踪
6. 报表统计

非功能需求：
1. 性能：竞价 RTT < 100ms
2. 可用性：99.99%
3. 扩展性：支持 QPS 从 1 万到 100 万
4. 安全性：防欺诈、防篡改
5. 可维护性：每周发布 3 次
```

### 2.3 容量估算

```
QPS 估算：
- DAU（日活用户）：1000 万
- 人均广告请求：10 次/天
- 峰值系数：5x（早晚高峰）
- 峰值 QPS = 1000 万 × 10 × 5 / 86400 ≈ 5787 QPS

数据量估算：
- 每日曝光日志：1000 万 × 10 = 1 亿条
- 每条日志 1KB
- 每日数据量：100GB
- 7 天保留：700GB
- 30 天保留：3TB

带宽估算：
- 请求大小：1KB
- 响应大小：2KB
- 峰值 QPS：6000
- 带宽 = 6000 × (1 + 2) KB/s ≈ 18 MB/s
```

## 第三部分：架构设计原则

### 3.1 SOLID 原则

```
S - Single Responsibility Principle（单一职责）
  → 一个类/模块只负责一件事

O - Open/Closed Principle（开闭原则）
  → 对扩展开放，对修改关闭

L - Liskov Substitution Principle（里氏替换）
  → 子类必须能够替换父类

I - Interface Segregation Principle（接口隔离）
  → 接口要小而专，不要大而全

D - Dependency Inversion Principle（依赖倒置）
  → 依赖抽象，不依赖具体实现
```

### 3.2 架构设计原则

```
1. 关注点分离（Separation of Concerns）
   → 不同职责分到不同模块

2. 高内聚低耦合（High Cohesion, Low Coupling）
   → 模块内紧密，模块间松散

3. 依赖倒置（Dependency Inversion）
   → 依赖抽象，不依赖具体

4. 开闭原则（Open/Closed）
   → 通过扩展而不是修改来改变行为

5. 迪米特法则（Law of Demeter）
   → 只和直接朋友通信

6. 里氏替换（Liskov Substitution）
   → 子类可以替换父类

7. 组合优于继承（Composition over Inheritance）
   → 优先使用组合而不是继承
```


## 第五部分：架构演进

### 5.1 架构演进路径

```
阶段 1：单体架构
  → 所有功能在一个应用中
  → 适合初创期，快速迭代

阶段 2：垂直拆分
  → 按业务拆分为多个应用
  → 用户服务、广告服务等

阶段 3：水平拆分（微服务）
  → 按领域拆分为多个服务
  → 竞价服务、预算服务、报表服务等

阶段 4：云原生
  → 容器化、Service Mesh、Serverless
  → 弹性伸缩、自动扩缩容

阶段 5：智能化
  → AI 驱动架构优化
  → 自动扩缩容、智能路由
```

### 5.2 单体到微服务的演进

```
单体架构：
┌─────────────────────────────────────────────────────┐
│                    Monolith                         │
│  ┌──────────┬──────────┬──────────┬──────────────┐  │
│  │ User     │ Ad       │ Bid      │ Report       │  │
│  │ Service  │ Service  │ Engine   │ Service      │  │
│  └──────────┴──────────┴──────────┴──────────────┘  │
└─────────────────────────────────────────────────────┘

微服务架构：
┌─────────────────────────────────────────────────────┐
│                   API Gateway                         │
├──────────┬──────────┬──────────┬────────────────────┤
│ User     │ Ad       │ Bid      │ Report             │
│ Service  │ Service  │ Engine   │ Service            │
│          │          │          │                    │
│ DB       │ DB       │ Redis    │ ClickHouse         │
└──────────┴──────────┴──────────┴────────────────────┘
```

### 5.3 演进中的挑战

```
1. 数据一致性：从单体数据库到分布式数据库
2. 服务通信：从方法调用到 RPC/HTTP
3. 部署复杂度：从一键部署到 K8s 编排
4. 监控难度：从单机日志到分布式追踪
5. 团队协调：从单人开发到多团队协作
```

### 6.2 高可用设计

```
1. 多可用区部署
   → 每个服务部署在多个 AZ
   → AZ 故障不影响服务

2. 主动-主动模式
   → 所有节点都提供服务
   → 负载均衡分发流量

3. 故障转移
   → 自动检测故障
   → 自动切换到健康节点

4. 降级策略
   → 非核心功能降级
   → 保证核心功能可用
```

```go
// 故障转移实现
type FailoverManager struct {
    healthyNodes []string
    unhealthyNodes []string
    mu sync.RWMutex
}

func (fm *FailoverManager) HealthCheck(node string) bool {
    // 定期健康检查
    resp, err := http.Get(fmt.Sprintf("http://%s/health", node))
    if err != nil {
        return false
    }
    defer resp.Body.Close()
    
    return resp.StatusCode == 200
}

func (fm *FailoverManager) MoveToHealthy(node string) {
    fm.mu.Lock()
    defer fm.mu.Unlock()
    
    // 从 unhealthy 移动到 healthy
    for i, n := range fm.unhealthyNodes {
        if n == node {
            fm.unhealthyNodes = append(fm.unhealthyNodes[:i], fm.unhealthyNodes[i+1:]...)
            fm.healthyNodes = append(fm.healthyNodes, node)
            break
        }
    }
}
```

### 6.3 性能优化

```
1. 缓存策略
   → LRU 缓存（本地）
   → Redis 缓存（分布式）
   → CDN 缓存（静态资源）

2. 异步处理
   → 消息队列（Kafka）
   → 事件驱动
   → 最终一致性

3. 批量处理
   → 批量写入（Flush）
   → 批量查询（Batch Get）
   → 批量上报（Batch Report）

4. 连接池
   → DB 连接池
   → HTTP 连接池
   → Redis 连接池
```

## 第七部分：生产排障案例

### 7.1 架构瓶颈分析

```
现象：竞价服务 P99 延迟从 50ms 飙升到 500ms

排查步骤：
1. 监控大盘 → 发现 Redis 连接数激增
2. pprof → 发现大量 goroutine 阻塞在 Redis 连接
3. 日志 → 发现大量超时错误
4. 根因：Redis 连接池耗尽

解决方案：
1. 增大连接池大小
2. 添加连接池监控
3. 使用本地缓存减少 Redis 查询
4. 添加熔断机制
```

### 7.2 架构演进问题

```
现象：微服务拆分后，接口响应变慢

排查：
1. 分布式追踪 → 发现跨服务调用延迟高
2. 网络延迟 → 服务间调用经过多个 hops
3. 序列化开销 → Protobuf vs JSON

解决方案：
1. 合并相关服务
2. 使用 gRPC 替代 HTTP/JSON
3. 添加服务网格优化路由
4. 异步化非关键调用
```
