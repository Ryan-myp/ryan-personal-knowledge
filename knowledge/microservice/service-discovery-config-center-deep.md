## 第一部分：服务发现

### 服务发现架构

```
服务发现核心组件：
┌─────────────────────────────────────────────────────────────────────┐
│ 1. 服务提供者                                                        │
│    ├── 注册服务：启动时注册                                         │
│    ├── 心跳保活：定期发送健康检查                                   │
│    └── 注销服务：优雅下线                                           │
│                                                                     │
│ 2. 服务消费者                                                        │
│    ├── 发现服务：查询可用实例                                       │
│    ├── 负载均衡：选择合适实例                                       │
│    └── 故障转移：自动切换健康实例                                   │
│                                                                     │
│ 3. 注册中心                                                          │
│    ├── 存储服务：服务实例信息                                       │
│    ├── 健康检查：监控实例状态                                       │
│    └── 通知机制：服务变更通知                                       │
└─────────────────────────────────────────────────────────────────────┘
```

## 第二部分：Consul 深度

### Consul 架构

```
Consul 架构组件：
┌─────────────────────────────────────────────────────────────────────┐
│ 1. Agent                                                              │
│    ├── Server Agent：参与 Raft 共识                                   │
│    ├── Client Agent：转发请求到 Server                               │
│    └── 健康检查：TCP/HTTP/TTL/Docker                                 │
│                                                                     │
│ 2. Datacenter                                                        │
│    ├── 服务注册：KV 存储                                              │
│    ├── DNS 接口：服务发现                                            │
│    └── HTTP API：RESTful 接口                                       │
│                                                                     │
│ 3. 成员管理                                                          │
│    ├── Gossip 协议：节点发现                                         │
│    ├── 故障检测：Ping/Pong                                           │
│    └── 视图一致性：Serf 广播                                         │
└─────────────────────────────────────────────────────────────────────┘
```

### Consul 服务发现

```
服务发现流程：
┌─────────────────────────────────────────────────────────────────────┐
│ 1. 服务注册                                                          │
│    ├── 启动时发送 PUT /v1/agent/service/register                     │
│    ├── 包含服务名、IP、端口、标签                                   │
│    └── Consul 存储到 KV 存储                                        │
│                                                                     │
│ 2. 健康检查                                                          │
│    ├── HTTP 检查：定期请求健康端点                                  │
│    ├── TCP 检查：建立 TCP 连接                                      │
│    └── TTL 检查：手动更新健康状态                                   │
│                                                                     │
│ 3. 服务发现                                                          │
│    ├── DNS 查询：service.consul                                    │
│    ├── HTTP API：GET /v1/health/service/{service}                  │
│    └── Catalog API：GET /v1/catalog/service/{service}              │
│                                                                     │
│ 4. 负载均衡                                                          │
│    ├── 随机选择                                                     │
│    ├── 轮询                                                         │
│    └── 加权                                                         │
└─────────────────────────────────────────────────────────────────────┘
```

## 第四部分：自测题

### Q1: 服务发现的三大组件？

**A**: 服务提供者、服务消费者、注册中心。

### Q2: Consul 的健康检查方式？

**A**: HTTP 检查、TCP 检查、TTL 检查、Docker 检查。

### Q3: Nacos 的配置模型？

**A**: Data ID（唯一标识）、Group（分组）、Namespace（环境隔离）。
