# 知识文档质量标准

> 标签: `#知识标准` `#源码级` `#quality-standards`

---

## 合格标准：四件套合一

完整知识文档必须包含以下全部要素，**缺少任何一项 = 不精**：

| 要素 | 说明 | 示例 |
|------|------|------|
| **1. 源码级深度** | 具体的数据结构定义（struct）、关键函数签名、核心算法逻辑、完整代码实现 | Go struct 定义含字段含义 |
| **2. ASCII 架构图/流程图** | 用 ╔╗╚╝║═╠╣╩╪╬ 等字符画的架构/流程图 | 系统架构图 |
| **3. 对比表格** | 与竞品/替代方案的对比 | Redis vs Memcached 对比 |
| **4. 实战排障/调优** | 真实生产场景的排查思路、调优参数 | Redis maxmemory 策略 |
| **5. 自测题** | 至少 3 个问题，答不上 = 没懂，每个含查看答案 | 折叠块含参考答案 |
| **6. 动手验证** | 可运行的代码示例 | Go 可运行示例 |

---

## 语言要求

- **技术示例代码必须用 Go**（用户团队使用 Go）
- ❌ 用 Java 写设计模式示例 = 错误
- ✅ 如果主题是 Java 生态（如 Spring、JVM 底层），可以用 Java
- 代码注释、变量名保持英文

---

## 文档结构模板

### 入门引导（5 分钟速览）
- 通俗解释：是什么、为什么、快速体验代码
- 架构图解：ASCII 架构/流程图
- 关键概念表格

### 源码级深度剖析
- 核心数据结构源码（struct 定义，含字段含义）
- 关键算法/流程源码（函数签名 + 核心逻辑注释）
- Token 消耗、错误处理

### 自测
- 3 个问题，答不上 = 没懂
- 每个问题包含查看答案的折叠块

### 动手验证
- 可运行的代码示例
- 观察要点

## 反模式清单

| ❌ 错误做法 | ✅ 正确做法 |
|-----------|-----------|
| 入门级名词解释（什么是 Redis、什么是 MVCC） | 源码级剖析（GMP调度器的源码实现） |
| 只给大纲/框架/目录结构，没有实际内容 | 一次性给完整的、可直接阅读的内容 |
| 先用浅内容再说深内容，反复迭代 | 一次性给有基础的人能看懂的粒度 |
| 只有代码没有图解 | 源码 + 图表 + 表格 + 自测 + 代码 |
| 内容随时间越来越浅 | 每次新增都要达到同等深度 |
| 空目录/空文件 | 要么删除，要么填充实质内容 |
| 目录名用连字符（my-topic） | 目录名用下划线（my_topic/） |

## 文件大小标准

| 类别 | 最低行数 | 推荐行数 |
|------|----------|----------|
| 技术深度文档 | 400 行 | 600-1000 行 |
| 系统架构文档 | 500 行 | 800-1200 行 |
| API 参考 | 200 行 | 300-500 行 |
| 入门/索引/导航 | 100 行 | 150-300 行 |
| README | 20 行 | 50-100 行 |
| 工具配置 | 不强制 | 按实际 |

## 目录组织原则

```
knowledge/
├── agent-ai/           ← AI Agent 技术（RAG/ReAct/多Agent）
├── ad-ads/             ← 广告系统（竞价/排序/数据分析）
├── advertising/        ← 广告平台 API 深度体系
├── architecture/       ← 架构设计（设计模式/DDD/CQRS）
├── architecture-patterns/ ← 架构模式总结
├── fullstack/          ← 全栈深入（Go/MySQL/Redis/ES/ClickHouse/大数据）
├── middleware/         ← 中间件（Kafka等）
├── tools/              ← 工具集成（微信读书API等）
└── growth-plan/        ← 成长路线图
```

- **不要创建空子目录**
- **按技术类别分类**，不要把所有技术混在一起
- **扁平化**：避免超过 2 层的嵌套目录
